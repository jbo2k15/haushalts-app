## Performance (für später, niedrige Priorität bei aktueller Haushaltsgröße)

- [ ] **`updateTrophyCache` lädt die komplette Completed-Historie in den Speicher** (`backend/src/services/scheduler.js`) — `prisma.taskLog.findMany({ where: { status: 'completed', … } })` ohne Datumsgrenze zieht jeden je erledigten Log in den Node-Prozess, um Trophäen neu zu berechnen (täglich um Mitternacht + bei jedem Serverstart). Einziges Muster, das mit der Gesamthistorie statt mit aktiven Daten wächst. Bei einem Haushalt über Jahre einige Tausend Zeilen — absolut noch harmlos, läuft selten, selektiert nur 2 Spalten. Trophäen sind kumulativ über alle Zeit, brauchen also grundsätzlich die Historie; saubere Lösung wäre eine inkrementelle Berechnung (nur neue Perioden), aber deutlich mehr Aufwand. Erst angehen, wenn die Historie spürbar groß wird.
- [ ] **`taskLog` hat keine Aufbewahrungsgrenze** — Tokens und Abfall-Tasks werden geprunt, `taskLog` wächst dagegen unbegrenzt (jede Erledigung/Verfall/Übersprung). Speist die Trophäen-Neuberechnung (oben) und die `/tasks/stats`-Aggregation. Bei eurem Volumen (grob ~10k Zeilen/Jahr) für SQLite trivial. Falls gewünscht: nächtlicher Cleanup-Job, der Logs älter als z. B. 1 Jahr löscht — aber Achtung, das würde die kumulativen Trophäen-Zahlen verfälschen, daher nur mit angepasster Trophäen-Logik umsetzbar. Vorerst bewusst offen.

## Barrierefreiheit (Accessibility-Review 2026-07-22, PWA-Fokus, noch nichts umgesetzt)

- [ ] **Icon-only Buttons ohne `aria-label`** — mehrere „✕"-Schließen-Buttons (`PageCarousel.jsx` Swipe-Tipp, `ReleaseNotesModal.jsx`, `Admin.jsx` Aufgabe-bearbeiten-Modal, `TaskRow.jsx` „Heute nicht nötig") haben nur das sichtbare Zeichen, keinen Accessible Name — Screenreader lesen "✕" oder gar nichts Sinnvolles vor. `title`-Attribute (bei zweien in `TaskRow.jsx` vorhanden) werden von vielen Screenreadern/auf Touch gar nicht vorgelesen. Fix: `aria-label="Schließen"` bzw. passenden Text ergänzen.
- [ ] **Drag & Drop in der Aufgabenverwaltung ist rein Maus-/Touch-bedienbar** — `Admin.jsx` registriert nur `MouseSensor`/`TouchSensor` bei `@dnd-kit`, keinen `KeyboardSensor`. Tastatur-Nutzer können Aufgaben aktuell gar nicht per Drag & Drop umsortieren. `@dnd-kit/core` bringt einen `KeyboardSensor` mit (Pfeiltasten + Leertaste zum Aufnehmen/Ablegen) — müsste ergänzt und die `SortableContext`/`sortableKeyboardCoordinates` entsprechend verdrahtet werden.
- [ ] **Kein Fokus-Trap in Modals** — Aufgabe-bearbeiten-Modal und „App schließen?"-Dialog fokussieren zwar automatisch das erste sinnvolle Element beim Öffnen (siehe letzte Änderung), aber Tab kann den Fokus trotzdem auf Elemente HINTER dem Modal verlassen, solange es offen ist. Für echte Modal-Semantik müsste Tab/Shift+Tab innerhalb des Dialogs zirkulieren (Fokus-Falle).
- [ ] **Touch-Ziele unter der empfohlenen Mindestgröße (44×44px)** — mehrere Icon-Buttons sind kleiner: „Zurücknehmen"/„Heute nicht nötig" in `TaskRow.jsx` (28×28px, `w-7 h-7`), Header-Menü-Toggle (36×36px, `w-9 h-9`), Zoom-Buttons in `Settings.jsx` (32×32px, `w-8 h-8`). Für eine Touch-first-PWA unterhalb gängiger Empfehlungen (Apple HIG/Material: 44-48px), auch wenn WCAG 2.5.5 selbst nur AAA-Stufe ist.
- [ ] **Keine `prefers-reduced-motion`-Behandlung** — Übergänge/Animationen (Embla-Swipe, `transition-colors`/`transition-transform` an diversen Stellen) laufen unabhängig von der Betriebssystem-Einstellung „Bewegung reduzieren". Betrifft potenziell empfindliche Nutzer (Vestibularstörungen).
- [ ] **Keine semantischen Landmarks** — Seiten nutzen durchgehend `<div>` statt `<main>`/`<nav>`/`<header>`; kein „Zum Inhalt springen"-Link vorhanden. Screenreader-Nutzer haben dadurch keine Sprungmarken zur groben Seitenstruktur.
- [ ] **Dynamische SSE-Updates werden nicht angekündigt** — ändert ein anderer Nutzer eine Aufgabe (Server-Sent Events, siehe `Home.jsx`), aktualisiert sich die Liste zwar visuell, aber ohne `aria-live`-Region bekommen Screenreader-Nutzer das nicht mit.
- [ ] **Farbkontrast nicht automatisiert geprüft** — dieser Review war eine Code-/Struktur-Analyse, kein visueller Kontrast-Check. Empfehlung: einmal mit einem automatisierten Tool (z. B. Lighthouse-Accessibility-Audit oder axe DevTools) gegen die laufende App prüfen, besonders Textfarben auf den orangen/farbigen Flächen (Badges, aktive Tabs) in Hell- und Dunkelmodus.

## Feature-Ideen (für später, noch nicht angefangen)

- [ ] **Mehrere Pausenzeiträume im Voraus planen** (Anforderungen geschärft am 2026-07-21, eigener Folgepunkt zum Pausenzeitraum-Feature oben, **noch nicht umgesetzt**) — Der Nutzer möchte sowohl mehrere individuelle Pausenzeiträume pro Aufgabe als auch mehrere globale Pausenzeiträume gleichzeitig im Voraus eintragen können (z. B. Weihnachtsferien UND Sommerferien schon jetzt beide hinterlegen). Die Architektur (eigene Tabellen `TaskPause`/`GlobalPause` statt Singleton/Task-Spalten) ist dafür bereits vorbereitet, kein neuer Datenmodell-Umbau nötig.
  - **Technische Lücke, die vor der Umsetzung behoben werden muss (kein Entscheidungsbedarf, nur Umsetzungshinweis):** `isPeriodFullyPaused` (Verfallen-Unterdrückung) ist bereits array-fähig für mehrere Zeiträume, aber `isPausedOnDay` (einfache Tagesprüfung für "heute ausblenden") nimmt aktuell nur genau EINEN individuellen und EINEN globalen Zeitraum entgegen — muss auf "gilt einer von mehreren Zeiträumen für heute?" umgestellt werden. Das betrifft **beide Seiten**: nicht nur die individuelle Pause, auch `getGlobalPause()` müsste zu `getGlobalPauses()` (Plural) werden, damit z. B. eine Weihnachts- UND eine Sommer-Global-Pause gleichzeitig in Tages- und Verfallen-Prüfung berücksichtigt werden. Ebenso löschen `setIndividualPause`/`setGlobalPause` aktuell alle bestehenden Zeilen vor dem Anlegen einer neuen (Ersetzen-Semantik) — müssen auf "hinzufügen" umgestellt werden, plus eine Möglichkeit, einzelne Zeiträume per ID gezielt zu löschen (statt nur "alle löschen").
  - **Export/Import muss diesmal wirklich überarbeitet werden (Korrektur einer früheren Annahme):** Die ursprüngliche Pausenzeitraum-Spezifikation ging davon aus, Export/Import "nimmt die neuen Pause-Felder automatisch mit" — das stimmt nur für genau einen Zeitraum pro Aufgabe (`pauseFrom`/`pauseTo` als einzelne Felder). Bei mehreren Zeiträumen braucht der Export ein Array von Zeiträumen pro Aufgabe statt zwei einzelner Felder — eine echte Format-Änderung, keine automatische Mitnahme mehr.
  - **Überlappung erlaubt, keine Validierung (entschieden 2026-07-21):** Mehrere sich überschneidende Zeiträume für dieselbe Aufgabe/global sind zulässig und werden nicht geprüft — die ODER-Verknüpfung kommt damit problemlos klar, das Ergebnis ("heute pausiert: ja/nein") bleibt identisch, eine Überlappungs-Validierung hätte keinen funktionalen Nutzen.
  - **Obergrenze 20 Zeiträume** pro Aufgabe bzw. global (entschieden 2026-07-21) — analog zu bestehenden Limits im Projekt (z. B. max. 200 Aufgaben beim Import), weit über dem realistischen Bedarf, verhindert aber eine ausufernde Liste.
  - **Admin-UI:** eigener Bereich pro Aufgabe außerhalb des normalen Bearbeiten-Formulars (entschieden 2026-07-21), z. B. über einen Link/Button in der Aufgabenzeile ("Pausen verwalten"), der eine eigene Listen-Ansicht mit Hinzufügen/Bearbeiten/Entfernen öffnet — hält das normale Bearbeiten-Formular unverändert kompakt, da Pausen ein Nebenaspekt sind. Analog für die globale Pause in der Verwaltung.
  - **Symbol-Vorrang in der Aufgabenliste (entschieden 2026-07-21):** Hat eine Aufgabe gleichzeitig einen aktiven UND einen zukünftig geplanten Zeitraum, zeigt `SortableTask.jsx` nur "⏸ Pausiert" (aktiv hat Vorrang) — "⏳ Pause geplant" erscheint nur, wenn gerade nichts aktiv läuft. Details zu allen Zeiträumen sind über den neuen "Pausen verwalten"-Bereich einsehbar.

- [ ] **Fairness-Transparenz in der Ruhmeshalle** (erweiterte Variante mit Gewichtung) — Recherche zu Erwachsenen-/Paar-Chore-Apps (evenus.app, FairShare, Chore Chores, FairChore) zeigt: die Ruhmeshalle optimiert aktuell auf Wettbewerb (Pokale, Sieger), nicht auf Fairness (wer trägt wie viel bei). Idee: zusätzliche Karte auf der Ruhmeshalle-Seite, unterhalb der bestehenden Legende:
  - Gestapelter Balken zeigt den prozentualen Anteil erledigter Aufgaben pro Person über ein rollierendes Fenster (z. B. 30 Tage)
  - Trend-Satz darunter, der einordnet, ob sich die Verteilung verbessert/verschlechtert hat ggü. dem Vormonat
  - Kleiner Wochenverlauf (gestapelte Mini-Balken pro Kalenderwoche), damit einzelne Ausreißerwochen (Urlaub, Krankheit) nicht als dauerhafte Schieflage fehlinterpretiert werden
  - Erweiterte Variante (statt reiner Zählung): jede Aufgabe bekommt ein optionales Schwierigkeits-/Aufwandsgewicht (z. B. 1 = Leicht, 2 = Mittel, 3 = Aufwändig), damit "Müll rausbringen" nicht gleich zählt wie "Bad putzen" — erfordert neues `difficulty`-Feld am Task-Modell (Migration, Default "Mittel" sowohl für neue als auch bestehende Aufgaben) plus Verwaltungs-UI zum Setzen des Werts (Auswahl Leicht/Mittel/Aufwändig im Bearbeiten-Formular, JSON-Export/Import nimmt das Feld automatisch mit)
  - **Berechnungsformel:** pro Person Summe aus (Anzahl Erledigungen × Schwierigkeitsgewicht der jeweiligen Aufgabe zum Erledigungszeitpunkt) über das Zeitfenster; die beiden (bzw. mehreren) Personensummen werden zueinander ins Verhältnis gesetzt (Anteil = eigene Summe / Summe aller). Jede einzelne Erledigung zählt separat (nicht nur "an dem Tag mindestens einmal gemacht"), nutzt also das bestehende Mehrfach-Erledigen-Feature korrekt aus.
  - **Trend-Vergleich zum Vormonat und Wochenverlauf** müssen beide nach derselben gewichteten Formel berechnet werden (gewichteter Anteil des vorherigen 30-Tage-Fensters bzw. je Kalenderwoche), nicht nach reiner Erledigungs-Anzahl — sonst sind aktueller und vorheriger Wert nicht vergleichbar. Der Trend-Satz sollte das auch explizit benennen ("gewichteter Anteil war X/Y"), damit nicht der Eindruck entsteht, es handle sich um eine reine Erledigungs-Anzahl.
  - **Schwierigkeitsgewicht wird beim Erledigen eingefroren** (Snapshot im Completion-Datensatz), nicht live aus dem aktuellen Task-Wert berechnet — eine nachträgliche Korrektur des Gewichts verändert also nicht rückwirkend die Statistik vergangener Zeiträume. Bewusste Entscheidung des Nutzers (2026-07-09), trotz des Nachteils, dass sich ein falsch gesetztes Anfangsgewicht später nicht rückwirkend korrigieren lässt.
  - **Zurückgenommene Erledigungen** (bestehendes Undo-Feature) müssen aus der Statistik verschwinden — ergibt sich automatisch, wenn die Fairness-Zahlen live aus den echten Completion-Datensätzen aggregiert werden statt separat mitgezählt.
  - **Sichtbarkeit:** globaler Ein/Aus-Schalter in der Verwaltung ("Fairness-Statistik anzeigen"), da Fairness-Zahlen ohne Kontext auch Konflikte auslösen können statt sie zu lösen. Bei 2 aktuellen Haushaltsmitgliedern (Design-Grenze: bis ca. 4–6 Personen sinnvoll, danach bräuchte der gestapelte Balken eine Listen-/Tabellen-Alternative).
  - **Farben neutral wählen** (z. B. Lila/Teal statt Grün/Koralle) — eine Grün/Rot-ähnliche Farbgebung würde unbewusst als Gut/Schlecht-Bewertung gelesen, obwohl nur eine neutrale Verteilung gezeigt werden soll.
  - **Urlaubsmodus verzerrt aktuell nicht** — Nutzer sind gemeinsam im Urlaub, kein Sonderfall für die Berechnung nötig.
  - **Aufgaben-Erledigen-UI bleibt bewusst unverändert** — der Schwierigkeitsgrad wird beim Abhaken nicht angezeigt, um "strategisches" Aufgaben-Picken (leichte Aufgaben bevorzugen) zu vermeiden.
  - Mockups wurden in einer Konversation mit Claude Code erstellt (2026-07-09, zuletzt aktualisiert): (1) Fairness-Karte mit gestapeltem Balken (Lila/Teal statt Grün/Koralle), Trend-Satz explizit als "gewichteter Anteil" beschriftet (war zunächst mehrdeutig als reine Anzahl lesbar, jetzt korrigiert), zwei Stat-Kärtchen "erledigt + Ø Schwierigkeit" pro Person, 4-Wochen-Verlauf als Mini-Balkendiagramm (zeigt ebenfalls den gewichteten Wochen-Anteil, nicht die Rohzahl); (2) Verwaltungs-Bereich mit Ein/Aus-Schalter "Fairness-Statistik anzeigen" plus Einstieg zum Pflegen der Schwierigkeitsgrade; (3) Aufgabenverwaltung — Aufgabenliste mit Schwierigkeits-Badge und Bearbeiten-Formular mit neuem Leicht/Mittel/Aufwändig-Feld (Default Mittel). Alle visuell im bestehenden App-Stil (abgerundete Karten, gleiche Farbpalette).
  - **Kompatibilität mit inzwischen gebauten Features (bestätigt 2026-07-21, kein Umsetzungsaufwand):** Pausenzeitraum verzerrt nicht — eine pausierte Aufgabe erzeugt während der Pause keine Erledigungen, analog zum bereits notierten Urlaubsmodus-Fall. Wetterabhängige Aufgaben verzerren ebenfalls nicht — `system-completed`-Einträge erzeugen bewusst keinen `TaskCompletion`-Datensatz, fließen also automatisch nicht in die Formel ein.
  - **Rundung (entschieden 2026-07-21):** ganze Prozent (z. B. "57%"), keine Nachkommastelle — würde bei kleinen Haushalten (2-3 Personen) eine Genauigkeit vortäuschen, die die Datenmenge nicht hergibt.
  - **Leerer Zustand bei Summe aller = 0 (entschieden 2026-07-21):** Hat im gesamten Zeitfenster noch niemand im Haushalt etwas erledigt, zeigt die Karte statt des Balkens einen Hinweistext ("Noch keine Erledigungen in diesem Zeitraum") — vermeidet die Division-durch-0 sauber und täuscht kein irreführendes "perfekt ausgeglichenes" 0%/0%-Patt vor.
  - **Listen-Alternative ab 5 Personen (entschieden 2026-07-21):** Ab 5 Haushaltsmitgliedern ersetzt eine einfache absteigend sortierte Liste den gestapelten Balken, Format je Zeile: "Name — 57% (18 gewichtete Punkte)". Keine neue Komponente nötig, nur eine alternative Darstellung derselben zugrundeliegenden Daten.

- [ ] **Mehrere Haushalte** (erste Grundsatzentscheidungen getroffen am 2026-07-21, weiterhin früh in der Schärfung — deutlich größer als die anderen Feature-Ideen, da praktisch jedes Datenmodell aktuell von genau einem Haushalt ausgeht) — **Ziel/Motivation (2026-07-21):** Der Nutzer möchte diese App befreundeten Familien bereitstellen, mit denselben Möglichkeiten wie seinem eigenen Haushalt heute, nur auf komplett getrennter Datengrundlage. Eckpunkte:
  - Getrennte Aufgaben und Statistiken pro Haushalt
  - Ein Nutzer kann Mitglied in mehreren Haushalten sein
  - Nutzer kann zwischen seinen Haushalten wechseln
  - Beitritt zu einem Haushalt nur per Einladung (kein offener Self-Signup pro Haushalt)
  - **Rollen-Scope (entschieden 2026-07-21):** Admin/Nutzer-Rolle ist pro Haushalt getrennt, nicht global am Account — jemand kann in Haushalt A Admin sein und in Haushalt B normaler Nutzer. Erfordert eine neue Mitgliedschafts-Tabelle (User × Haushalt × Rolle) statt der Rolle direkt am `User`-Modell.
  - **Einladungs-Mechanismus (entschieden 2026-07-21):** Beitrittscode pro Haushalt (kein E-Mail-Einladungslink) — Nutzer tritt mit dem Code bei, muss aber anschließend noch von einem Admin des Haushalts freigeschaltet werden (gleiches Freischaltungs-Prinzip wie bei der bestehenden Registrierung, nur zusätzlich an einen Haushalt gebunden statt global). Der erste Nutzer eines neu angelegten Haushalts wird automatisch dessen Admin (kein Henne-Ei-Problem beim Haushalt-Anlegen).
  - **Statistik-/Trophäen-Scope (entschieden 2026-07-21):** pro Haushalt getrennt, nicht global über alle Haushalte eines Nutzers summiert — Trophäen/Fairness sind nur im Vergleich zu den Mitbewohnern des jeweiligen Haushalts sinnvoll. Die Ruhmeshalle zeigt immer nur den gerade aktiven Haushalt.
  - **Aktiver Haushalt (entschieden 2026-07-21):** explizite Auswahl über ein Dropdown/Umschalter im Header-Menü, serverseitig am Account gespeichert (nicht nur lokal wie das Theme) — konsistent über alle Geräte hinweg, und Push-Benachrichtigungs-Jobs wissen dadurch eindeutig, für welchen Haushalt sie gerade zählen sollen.
  - **Wer darf einen Haushalt anlegen (entschieden 2026-07-21):** jeder registrierte Nutzer kann selbst über einen "Haushalt erstellen"-Button einen neuen Haushalt gründen (voll Self-Service, nicht auf den Betreiber beschränkt) und wird automatisch dessen erster Admin.
  - **Betreiber-Zugriff (entschieden 2026-07-21):** echte Isolation, kein Platform-Admin-Zugriff über alle Haushalte hinweg — der Betreiber sieht nur Haushalte, in die er selbst als normales Mitglied eingeladen wurde, genau wie jeder andere Nutzer. Bei Support-Bedarf greift er server-/datenbankseitig zu, nicht über eine App-Funktion.
  - **Technischer Fund, ergibt sich zwingend aus dem Ziel (2026-07-21):** `WEATHER_LAT`/`WEATHER_LON` (Wetterabhängige Aufgaben) und `WASTE_ICAL_URL` (Abfallkalender) sind aktuell serverweite ENV-Variablen, obwohl sie eigentlich haushaltsspezifisch sind — befreundete Familien an einem anderen Ort/mit anderem Entsorger brauchen eigene Werte. Beide müssen von ENV-Variablen zu einem Datenbank-Feld pro Haushalt wandern, inkl. Verwaltungs-UI zum Setzen. Instanzweite Werte (`JWT_SECRET`, VAPID-Schlüssel, SMTP-Zugang, `FRONTEND_URL`) bleiben dagegen zurecht global.
  - **Hosting/Skalierung (Randbedingung, präzisiert 2026-07-21):** Bleibt vorerst beim eigenen Server (Proxmox), erwartete Größenordnung ist eine Handvoll neuer Haushalte im Familien-/Freundeskreis — kein Anlass für einen Datenbank-Wechsel (SQLite reicht, Prisma abstrahiert den Datasource-Provider ohnehin weitgehend, falls später doch nötig). Perspektivisch soll aber ein Wechsel zu einem Hosting-Dienstleister architektonisch nicht blockiert sein: Die Mandantentrennung muss deshalb konsequent auf **Anwendungsebene** durchgesetzt werden (jede Query nach `householdId` gescoped), nicht durch physische/Infrastruktur-Trennung — passt zur bestehenden Entscheidung, `WEATHER_LAT`/`WASTE_ICAL_URL` pro Haushalt in die Datenbank statt in Server-ENV zu legen, da ein Dienstleister keinen ENV-Zugriff pro Kunde böte.
  - **Beitrittscode-Details (entschieden 2026-07-22):** kurzer alphanumerischer Code (6-8 Zeichen, ohne verwechselbare Zeichen wie 0/O oder 1/I/l) — leicht per WhatsApp/SMS/mündlich weiterzugeben, zusätzlich als klickbarer Link mit vorausgefülltem Code teilbar (z. B. `/join?code=A3F9K2`). **Einmalig statt wiederverwendbar:** ein Code gilt für genau eine einzuladende Person — der Admin generiert für jedes weitere Haushaltsmitglied einen eigenen neuen Code, kein geteilter Code für mehrere Beitritte. **Befristet auf 10 Tage** ab Erzeugung, danach ungültig; muss der Admin neu erstellen und erneut teilen (analog zum bestehenden Passwort-Reset-Token-Muster, nur mit längerer Gültigkeit passend zum Anwendungsfall "Familie tritt evtl. erst nach ein paar Tagen bei").
  - **Weiterhin zu klären, bevor die Umsetzung starten kann:** genaues Datenmodell für die Migration bestehender Nutzer/Aufgaben (vermutlich: ein impliziter Default-Haushalt, dem alle bestehenden Daten beim Umstieg zugeordnet werden), Auswirkung auf Push-Benachrichtigungen im Detail (eigene Erinnerungszeiten pro Haushalt statt einer globalen `NotificationSettings`-Zeile), Verwaltungs-UI für Haushalt-Erstellung/-Verwaltung/-Verlassen (inkl. Übersicht/Widerruf offener Beitrittscodes).
